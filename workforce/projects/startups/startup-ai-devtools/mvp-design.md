# MVP Design: SentinelCode

## Overview

The SentinelCode MVP is a focused, functional product that demonstrates core value propositions: AI-generated code security validation, team context awareness, and technical debt tracking. The MVP targets teams of 5-50 developers using GitHub Copilot or Cursor.

---

## MVP Scope

### In Scope (MVP)
1. Real-time security scanning of AI-generated code
2. VS Code extension for IDE integration
3. GitHub App for PR review automation
4. Basic team context learning
5. OWASP Top 10 vulnerability detection
6. Simple dashboard for metrics
7. Slack notifications

### Out of Scope (Post-MVP)
1. Advanced ML-based context learning
2. Custom security rule engine
3. On-premise deployment
4. Multi-platform IDE support (JetBrains, etc.)
5. Advanced analytics and reporting
6. Knowledge graph and documentation
7. Jira/Linear integration
8. API for custom integrations

## Feature Specifications

### Feature 1: Real-Time Security Scanner (Core)

**User Story**: As a developer, I want AI-generated code to be automatically scanned for security vulnerabilities so I can catch issues before committing.

**Acceptance Criteria**:
- [ ] Scans code in real-time as it's generated by AI
- [ ] Detects OWASP Top 10 vulnerabilities
- [ ] Shows inline warnings in IDE
- [ ] Provides fix suggestions
- [ ] False positive rate < 10%

**Technical Design**:
```typescript
// Security Scanner Service
interface SecurityScanner {
  scan(code: string, language: string, context: CodeContext): Promise<ScanResult>;
}

interface ScanResult {
  vulnerabilities: Vulnerability[];
  confidence: number;
  suggestedFix?: string;
}

interface Vulnerability {
  ruleId: string;
  severity: 'critical' | 'high' | 'medium' | 'low';
  line: number;
  column: number;
  message: string;
  cweId?: string;
  owaspCategory?: string;
}
```

**Implementation**:
- Use Semgrep for static analysis (open-source, extensible)
- Custom rules for AI-specific vulnerability patterns
- Tree-sitter for AST parsing
- WebSocket for real-time communication

### Feature 2: VS Code Extension (Primary Interface)

**User Story**: As a developer, I want security feedback directly in my IDE so I don't need to switch contexts.

**Acceptance Criteria**:
- [ ] One-click installation from VS Code marketplace
- [ ] Decorates code with security warnings
- [ ] Hover tooltips with vulnerability details
- [ ] Quick fixes via code actions
- [ ] Status bar with security score
- [ ] Settings for customization

**Technical Design**:
```typescript
// Extension Architecture
class SentinelCodeExtension {
  private scanner: SecurityScanner;
  private decorationManager: DecorationManager;
  private diagnosticCollection: vscode.DiagnosticCollection;

  async onDocumentChange(document: vscode.TextDocument) {
    const code = document.getText();
    const isAIGenerated = await this.detectAIGenerated(code);
    
    if (isAIGenerated) {
      const result = await this.scanner.scan(code, document.languageId);
      this.showDiagnostics(document, result);
    }
  }

  private async detectAIGenerated(code: string): Promise<boolean> {
    // Detect Copilot/Cursor comments or patterns
    return code.includes('// Generated by Copilot') ||
           code.includes('// Generated by Cursor') ||
           await this.heuristicDetection(code);
  }
}
```

**UI Components**:
- Code decorations (squiggly lines, gutter icons)
- Hover panels with vulnerability details
- Command palette integration
- Sidebar panel with security overview

### Feature 3: GitHub App Integration (CI/CD Gate)

**User Story**: As a team lead, I want AI-generated code to be automatically reviewed in pull requests so security issues don't reach production.

**Acceptance Criteria**:
- [ ] Automatically reviews PRs with AI-generated code
- [ ] Posts comments on security issues
- [ ] Blocks PRs with critical vulnerabilities (configurable)
- [ ] Provides summary report
- [ ] Integrates with GitHub Checks API

**Technical Design**:
```typescript
// GitHub App Webhook Handler
class GitHubAppHandler {
  async onPullRequestOpened(event: PullRequestEvent) {
    const files = await this.getChangedFiles(event);
    const aiGeneratedFiles = await this.filterAIGenerated(files);
    
    const results = await Promise.all(
      aiGeneratedFiles.map(file => this.scanner.scanFile(file))
    );

    await this.postReviewComments(event, results);
    await this.updateCheckRun(event, results);
  }

  private async postReviewComments(
    event: PullRequestEvent, 
    results: ScanResult[]
  ) {
    for (const result of results) {
      for (const vuln of result.vulnerabilities) {
        await this.github.rest.pulls.createReviewComment({
          owner: event.repository.owner.login,
          repo: event.repository.name,
          pull_number: event.pull_request.number,
          commit_id: event.pull_request.head.sha,
          path: result.filePath,
          line: vuln.line,
          body: this.formatComment(vuln)
        });
      }
    }
  }
}
```

### Feature 4: Team Context Learning (Basic)

**User Story**: As a team member, I want SentinelCode to learn our coding patterns so it can provide context-aware suggestions.

**Acceptance Criteria**:
- [ ] Analyzes existing codebase for patterns
- [ ] Learns naming conventions
- [ ] Identifies preferred libraries/frameworks
- [ ] Detects architectural patterns
- [ ] Uses context to reduce false positives

**Technical Design**:
```typescript
// Context Learning Engine
class ContextEngine {
  private vectorStore: VectorStore;
  private patternMatcher: PatternMatcher;

  async learnFromRepository(repo: Repository) {
    const files = await repo.getAllFiles();
    
    // Extract patterns
    const patterns = await Promise.all([
      this.extractNamingConventions(files),
      this.extractImportPatterns(files),
      this.extractArchitecturalPatterns(files)
    ]);

    // Store in vector DB
    await this.vectorStore.store(patterns);
  }

  async getRelevantContext(code: string): Promise<Context> {
    const embedding = await this.embed(code);
    return this.vectorStore.similaritySearch(embedding, 5);
  }
}
```

## Database Schema

### Core Tables

```sql
-- Organizations (multi-tenant)
CREATE TABLE organizations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  slug VARCHAR(255) UNIQUE NOT NULL,
  plan VARCHAR(50) DEFAULT 'free',
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Users
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(255) UNIQUE NOT NULL,
  name VARCHAR(255),
  organization_id UUID REFERENCES organizations(id),
  role VARCHAR(50) DEFAULT 'member',
  created_at TIMESTAMP DEFAULT NOW()
);

-- Repositories
CREATE TABLE repositories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID REFERENCES organizations(id),
  github_id BIGINT UNIQUE,
  name VARCHAR(255) NOT NULL,
  full_name VARCHAR(255) NOT NULL,
  default_branch VARCHAR(255) DEFAULT 'main',
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Security Scans
CREATE TABLE security_scans (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  repository_id UUID REFERENCES repositories(id),
  commit_sha VARCHAR(40),
  branch VARCHAR(255),
  trigger_type VARCHAR(50), -- 'ide', 'pr', 'manual'
  status VARCHAR(50) DEFAULT 'pending',
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  total_issues INTEGER DEFAULT 0,
  critical_count INTEGER DEFAULT 0,
  high_count INTEGER DEFAULT 0,
  medium_count INTEGER DEFAULT 0,
  low_count INTEGER DEFAULT 0
);

-- Vulnerabilities
CREATE TABLE vulnerabilities (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scan_id UUID REFERENCES security_scans(id),
  rule_id VARCHAR(255) NOT NULL,
  severity VARCHAR(20) NOT NULL,
  file_path TEXT NOT NULL,
  line_start INTEGER,
  line_end INTEGER,
  column_start INTEGER,
  column_end INTEGER,
  message TEXT NOT NULL,
  code_snippet TEXT,
  suggested_fix TEXT,
  cwe_id VARCHAR(20),
  owasp_category VARCHAR(50),
  status VARCHAR(50) DEFAULT 'open',
  created_at TIMESTAMP DEFAULT NOW()
);

-- Code Patterns (for context learning)
CREATE TABLE code_patterns (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  repository_id UUID REFERENCES repositories(id),
  pattern_type VARCHAR(50), -- 'naming', 'import', 'architecture'
  pattern_data JSONB NOT NULL,
  confidence FLOAT,
  occurrence_count INTEGER DEFAULT 1,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Metrics (time-series)
CREATE TABLE metrics (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID REFERENCES organizations(id),
  repository_id UUID REFERENCES repositories(id),
  metric_type VARCHAR(50) NOT NULL,
  metric_value FLOAT NOT NULL,
  recorded_at TIMESTAMP DEFAULT NOW()
);
```

## Security Considerations

### Data Security
- All code analysis happens in isolated ephemeral containers
- No permanent storage of customer code
- Vector embeddings stored separately from raw code
- Encryption at rest (AES-256) and in transit (TLS 1.3)

### Access Control
- Row-level security in PostgreSQL
- Organization-scoped resources
- Role-based permissions (admin, member, viewer)
- GitHub App minimal permissions model

### Compliance
- SOC 2 Type II roadmap (Year 2)
- GDPR compliance for EU customers
- Option for on-premise deployment (Enterprise)

### A. Security Rules Reference

```yaml
# Example custom rules for AI-generated code
rules:
  - id: ai-hardcoded-secret
    pattern-either:
      - pattern: const API_KEY = "..."
      - pattern: password = "..."
    message: "Potential hardcoded secret in AI-generated code"
    severity: ERROR

  - id: ai-sql-injection
    pattern: |
      $QUERY = "..." + $USER_INPUT + "..."
      ...
      execute($QUERY)
    message: "Potential SQL injection in AI-generated code"
    severity: ERROR
```
